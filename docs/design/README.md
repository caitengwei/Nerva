# Nerva 设计与测试文档

更新时间：2026-03-12

## 这组文档解决什么问题

Nerva 的代码量在增长，模块间的依赖关系不是看几个文件就能理清的。这组中文文档面向两类场景：

- **初次接触**：需要快速建立"请求怎么穿过系统、各层为什么这么分"的全貌认知。
- **日常开发**：需要确认改动的影响面、选择最小回归组合、理解性能指标的含义。

所有内容基于当前仓库代码（`src/nerva/`、`tests/`、`scripts/bench/`）整理。如果文档与实际运行不一致，以代码和最新回归结果为准。

## 建议阅读路径

### 初次接触 Nerva

1. **架构设计** §1-§4 — 理解 Nerva 的出发点、四个核心设计决策和请求生命周期。
2. **功能设计** §1-§2 — 用具体示例看模块分工，深入 DSL 层的 trace 机制。
3. **本地测试指南** §2-§3 — 能在本地稳定跑通 lint + type check + 测试。

### 准备改功能

1. **功能设计**中定位目标模块 — 看它的数据结构、对外接口和"改动注意"。
2. **架构设计** §5 执行模型 — 确认改动是否影响并发、故障处理或资源控制。
3. **本地测试指南** §8 — 按改动类型选最小回归组合。

### 准备做性能优化

1. **架构设计** §2.2 和 §5 — IPC 延迟数据、并发模型和资源控制闸门。
2. **性能测试指南** — 完整走一遍执行步骤，确保口径一致后再跟历史数据对比。

## 协作约定

- 功能行为有变化时，文档要跟着改。不要只改代码。
- 文档里写清"为什么这么做"，不只写"怎么做"。
- 涉及判断的结论给可复现证据（代码路径、命令、结果目录）。
