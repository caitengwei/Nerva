更新时间：2026-03-18

## 1. 设计目标

### 1.1 定位

单机多 GPU 的高性能多模型推理服务框架，定位为 Triton Inference Server 的 Python-first 替代。

### 1.2 核心目标（按优先级排序）

1. **多模型 DAG 编排** — 支持条件分支、并行、菱形依赖等复杂拓扑，pipeline 即 Python 函数。
2. **低延迟 + 高吞吐** — 单机极致性能，先把延迟和吞吐做到位再考虑分布式。
3. **进程隔离** — 每个模型独立 Worker 进程，故障不扩散、资源可控、绕开 GIL。
4. **Python-first** — 模型定义、pipeline 编排、服务配置全 Python，零 YAML/C++ 插件。
5. **统一后端抽象** — 插件式 Backend（PyTorch / vLLM），用户只需实现 `load()` + `infer()`。

### 1.3 目标用户

希望将模型部署到生产环境、但不想处理复杂基础设施细节的算法工程师。

### 1.4 MVP 边界

**MVP 不做**（避免引入不必要的复杂度）：

- 分布式集群部署（lock file + PID 检查有单机隐含假设）
- 流式推理（帧头已预留字段，当前只支持 unary）
- 运行时动态加卸载模型
- YAML 声明式配置
- Trace 的完整 Python 语义（不支持循环、副作用、运行时条件）
- 请求间优先级调度（EDF 等，当前 FIFO）
- 优雅 drain（SIGTERM 后直接关闭，不等待在途请求）
- Worker 侧入站背压（背压职责完全在 Master 侧 Batcher）

### 1.5 分维度目标

#### DAG

| 目标 | 状态 | 说明 |
|------|------|------|
| 任意 DAG 拓扑（线性/并行/菱形/条件/嵌套） | ✅ | trace 构图 + 显式 DAG 互补 |
| 图结构静态不变 | ✅ 设计约束 | trace 时确定，运行时不可变（PrecomputedGraph 冻结为 tuple） |
| 无依赖节点自然并行 | ✅ | 事件驱动调度，无需用户标注并行性 |
| 图分析开销摊销到启动期 | ✅ | PrecomputedGraph 每 pipeline 计算一次 |
| 图优化 pass | ❌ 后续迭代 | 自动并行化、设备亲和调度、死节点消除——设计文档提及但未落地 |

#### 调度

| 目标 | 状态 | 说明 |
|------|------|------|
| 请求内 DAG 并发 | ✅ | 事件驱动，in-degree table + done_queue |
| 请求间并发 | ✅ | 每请求独立 Executor 实例，互不阻塞 |
| Deadline 全链路贯穿 | ✅ | RPC 入口转 TTL → Batcher 检查 → Worker 超时 |
| Batching 对调度层透明 | ✅ | DynamicBatcher 实现 InferableProxy |
| 请求间优先级（EDF） | ❌ MVP 不做 | 当前 FIFO；后续迭代可升级 |
| Worker 侧入站背压 | ❌ MVP 不做 | 背压职责在 Master Batcher；后续迭代评估 Worker 侧限流 |

#### 进程级分布式

| 目标 | 状态 | 说明 |
|------|------|------|
| 每 model 独立 Worker 进程 | ✅ | GIL 无关 + 故障隔离 + 资源可控 |
| 多 uvicorn worker 共享 Nerva Worker | ✅ | spawn-once lock file + refcount |
| 单节点可配多实例 | ✅ | MultiInstanceProxy round-robin |
| Worker 崩溃自动重启 | ✅ | 退避策略 1s→16s，最多 5 次 |
| 孤儿进程防护 | ✅ | 父进程 watchdog（ppid 检测 + SIGTERM） |
| 优雅 drain | ❌ MVP 不做 | 依赖 uvicorn graceful shutdown；后续迭代实现 drain 阶段 |
| 跨机器 Worker | ❌ MVP 不做 | 沿现有进程边界外推，不重写核心调度 |

#### 性能

| 目标 | 状态 | 说明 |
|------|------|------|
| 框架开销 p50 < 10ms（3-stage pipeline） | ✅ 实测 ~8ms | 含 RPC body_read + IPC + 调度 |
| 并发线性扩展 | ✅ | C1→C16 QPS 线性，无退化 |
| 序列化开销可忽略 | ✅ | msgpack 全链路 < 0.1ms |
| IPC 大 payload 近零拷贝 | ✅ | >8KB 走 SHM，memoryview 解包 |
| rpc_body_read p99 尾延迟 | ⚠️ 已识别 | p99=8.3ms，asyncio 调度抖动 |
| ZMQ ipc_transport p99 尾延迟 | ⚠️ 已识别 | mm_postprocess p99=15ms |

## 2. 架构原则

### 2.1 多进程解耦

> 每个 `model()` 对应独立 Worker 进程，不共享 Python 运行时。

- **依据**：ADR-002, 架构设计 §2.1
- **代价**：IPC 开销（实测 per-stage p50 ~0.75ms）
- **收益**：GIL 无关 + 故障隔离 + 资源可控
- **约束**：不针对 mock artifact 改架构（memory: feedback_arch_principles）；mock 场景应贴近真实

### 2.2 双通道 IPC

> ZeroMQ 控制通道 + SHM 数据通道，按 payload 大小自动选路。

- **依据**：架构设计 §2.2, S1 spike 实测数据
- **规则**：≤8KB inline 进 ZeroMQ 消息，>8KB 写入 SHM 只传 descriptor
- **选型理由**：ZeroMQ ROUTER/DEALER 提供原生消息边界和心跳检测，延迟与原始 UDS 同量级

### 2.3 Executor 透明

> Executor 只认 `InferableProxy` 协议，不感知底层是单实例、多实例、Batcher 还是 mock。

- **依据**：ADR-002, multi-instance 设计
- **约束**：所有 Worker 层变更（多实例、batching、负载均衡）必须通过 InferableProxy 接口暴露，Executor 零改动

### 2.4 Pipeline 与 Serving 分离

> `Graph` 可独立执行，不依赖 HTTP 层。`serve()` 是部署层，不是 transform。

- **依据**：Design Review #9
- **收益**：单元测试和嵌入式使用不需要启动服务

### 2.5 trace + 显式 DAG 互补

> 受限 trace 降低常见场景的样板代码，显式 DAG 保证复杂场景的可预测性。

- **依据**：Design Review #4, S2 spike
- **trace 受限子集**：只捕获 ModelHandle.__call__ + cond/parallel + 简单表达式
- **不支持的模式抛明确错误**，引导用户回退到显式 DAG

## 3. DAG 原则

### 3.1 图静态不变

> trace 时确定图结构（节点、边、输入策略），运行时只读。

- **实现**：PrecomputedGraph 将邻接表、in-degree、source_nodes、输入策略全部冻结为 tuple
- **约束**：不支持运行时图变更、动态循环、数据依赖的拓扑变化
- **理由**：静态图可做一次性分析和优化，per-request 开销最小化

### 3.2 事件驱动调度

> 节点完成 → 后继入度减 1 → 入度为 0 立即调度。自然最大化并行度。

- **实现**：in-degree table + done_queue（asyncio.Queue）+ create_task
- **不是串行遍历拓扑序**：同层无依赖节点自然并行，无需用户标注
- **单线程安全**：done_queue 消费者是单个 asyncio 协程，无锁

### 3.3 sub-graph 隔离调度，共享资源

> cond/parallel 的每个分支创建子 Executor，共享 WorkerProxy 和顶层 InferContext。

- **parallel**：所有分支 `asyncio.gather()` 并发执行，结果按 index 存储
- **cond**：根据 predicate 选一条分支执行
- **关键约束**：子图是独立的调度单元，但不是独立的资源单元——共享 Worker 进程和连接

### 3.4 输入组装策略静态确定

> 4 种输入模式在图构建时预计算，运行时零分支判断。

| 策略 | 条件 | 行为 |
|------|------|------|
| SOURCE | 无入边 | 透传 pipeline 原始输入 |
| SINGLE_PASSTHROUGH | 单入边，无 dst_input_key | 透传上游输出（可含 field_path 提取） |
| KEYED_DICT | 多入边，每条都有 dst_input_key | 按 key 组装 dict |
| MULTI_LIST | 其他情况 | 按入边顺序组装 list |

### 3.5 控制流分支的闭包捕获语义

> cond/parallel 分支通过闭包捕获上游 Proxy 表达依赖，子图的边必须正确指向父图节点。

- **机制**：lambda 内引用的 Proxy 在 trace 时被 `_extract_proxy_edges()` 解析为跨图 Edge
- **高风险区域**（R-PH2-PROXY-CAPTURE）：分支 lambda 捕获的 Proxy 如果是上游节点输出（非 pipeline 输入），子图边的 `src` 必须指向父图中的正确节点
- **必测用例**：
  ```python
  out = a(x); cond(out["flag"], lambda: b(out), lambda: c(out))
  out = a(x); parallel(lambda: b(out), lambda: c(out))
  ```
- **断言要求**：不仅断言"不报错"，还要断言分支节点的输入值和输出值都正确

### 3.6 Edge 语义双向约定

> `src_field_path` 决定"从上游取什么"，`dst_input_key` 决定"放到下游哪里"。

- `resolve_field_path(output, edge.src_field_path)` 沿路径逐层取值
- 组装到 `inputs[edge.dst_input_key]`
- **约束**：两者的语义必须保持一致，对不上会在运行时拿错数据

## 4. 调度原则

### 4.1 每请求独立 Executor

> 请求间零共享状态。

- 每个请求创建独立的 Executor 实例，拥有自己的 in_degree 副本、completed dict、done_queue
- **代价**：per-request 分配 ~6 us（完整请求路径含 InferContext×3、IPC msg×6、timing×4 等全部对象）。占框架 CPU 开销的 0.14%（6 us / 4300 us）
- **收益**：代码简单、请求间完全隔离、无共享状态管理复杂度。asyncio 单线程模型下本不需要锁，隔离的真实价值在于消除共享状态 bug
- **调度开销的真正来源**：实测 `executor_scheduler` p50 = 510 us，其中 `asyncio.create_task` 占 ~100 us（每个 ~33 us × 3 节点），是 per-request 分配的 16 倍。对象分配不是瓶颈，task 创建才是
- **GC 影响**：100K 请求实测分配 p99 = 8 us，p99.9 = 26 us，GC gen0 收集 0 次。Python 短生命周期小对象回收高效，不会造成 p99 抖动
- **QPS 安全余量**：以模型推理服务 QPS 范围（几百到低千级），对象分配占单核 CPU < 1%。QPS 需达到 ~16,000（单请求 e2e < 1ms，纯 CPU 微服务场景）才会使分配成为 notable 开销
- **池化评估**：Executor 池化最多省 ~0.7 us/req，且引入状态泄漏风险（reset 遗漏字段 → 请求间数据串扰）。asyncio.Queue 无法 reset，必须重建，抵消池化主要收益。**不做池化**
- **后续优化方向**：对线性链拓扑（每个节点最多一个前驱一个后继）走 sequential await 快速路径，跳过 create_task + Queue 机制，可将调度开销从 ~115 us 降至 ~0 us。此优化在 PrecomputedGraph 阶段检测，对 API 透明
- **待观察**：QPS 显著提升后（如 GPU 推理延迟大幅降低导致 QPS 进入万级），需重新评估对象分配和 GC 的影响

### 4.2 Batcher 是 InferableProxy 的装饰器

> 对 Executor 完全透明，可选叠加。

- DynamicBatcher 包装任意 inner proxy，实现 InferableProxy
- 组图时透明：BatchConfig 通过 `model()` 参数声明
- vLLM 节点旁路 Batcher（vLLM 自带 continuous batching）
- **批内并发**：一个 batch 内的请求通过 `asyncio.gather()` 并发提交到 inner proxy（MVP 不做 tensor-level batching）

### 4.3 Deadline 三层守卫

> 每层独立检查，任何一层拦截都会快速失败。

| 层 | 时机 | 行为 |
|----|------|------|
| Batcher 入队 | 入队前 | 残余 TTL < 阈值 → 立即 DEADLINE_EXCEEDED |
| Batcher 出批 | flush 时 | 已过期请求从批次中过滤，future 设异常 |
| Worker 执行 | infer 调用 | `asyncio.wait_for(timeout=deadline_ms/1000)` |

Deadline 从请求进入 RPC handler 开始计时（绝对 epoch_ms 转相对 TTL），batch 等待时间计入 deadline。

### 4.4 FIFO 排队，无优先级

> MVP 不做 EDF 排序，同优先级下 FIFO 公平。

- **当前实现**：`asyncio.Queue`（FIFO）
- **设计文档 §6.3 提到的 EDF**：列入后续迭代
- **理由**：FIFO 实现简单、行为可预测；优先级调度增加排队数据结构复杂度，MVP 阶段收益不明显

### 4.5 背压职责在 Master

> Worker 信任 Master 的限流，自身不做入站排队限制。

- **Master 侧**：Batcher `queue_capacity`（默认 2048）+ `queue_timeout_ms` 限制排队深度和时间
- **Worker 侧**：对所有 INFER_SUBMIT 立即 `create_task`，无入站限流
- **MVP 有意选择**：简化 Worker 实现，避免 Worker 侧引入额外排队逻辑
- **后续迭代**：评估 Worker 侧轻量限流（如信号量）的必要性，特别是无 Batcher 场景下的高并发保护

## 5. 进程模型原则

### 5.1 Master 只调度，不推理

> Master 进程 CPU 开销仅来自 asyncio 调度 + msgpack 编解码 + ZeroMQ 通信。推理计算严格在 Worker 中。

- **约束**：Master 内不得出现 `model.infer()` 调用或任何 GPU 操作
- **演进方向**：当 Master CPU 成为瓶颈时，热路径 offload 到 Rust/C++ 线程池（NumPy 模式）

### 5.2 Worker 单模型，多实例横向扩

> 纵向扩（一个 Worker 装多模型）不做；横向扩（同模型多 Worker 进程）通过 instances 参数。

- **依据**：ADR-002
- **instances 是部署配置，不是图结构**——放在 `model()` 声明中，与 `backend`、`device`、`async_infer` 同层
- **MultiInstanceProxy**：round-robin 无锁 O(1) 分发，对 Executor 透明
- **Revisit 触发**：IPC 开销在多模型 pipeline 中占 e2e >20%

### 5.3 spawn-once + refcount = 多 uvicorn 安全共享

> 第一个 uvicorn worker 原子创建 lock → spawn Worker；后续 uvicorn worker 直连。

- **原子锁**：`O_CREAT | O_EXCL` 创建 lock file，写入 spawner PID
- **Refcount 文件**：`fcntl.flock` + read/write，跟踪连接数
- **关闭语义**：只有 refcount→0 时才 SHUTDOWN Worker 进程 + kill
- **陈旧检测**：proxy.start() 超时 → 删除 socket + refcount → 重新尝试 spawn

### 5.4 async_infer 决定 Worker 内并发模型

> IO-bound 模型用 async 多路复用，compute-bound 模型用线程池隔离。两者的性能特征和瓶颈模式完全不同。

| async_infer | Worker 内并发模型 | 适用场景 | 性能特征 |
|-------------|------------------|---------|---------|
| `True` | 事件循环内 `await backend.infer()` | IO-bound（网络调用、asyncio.sleep mock） | 单 Worker 可多路复用大量请求，瓶颈在 IO |
| `False` | 线程池 `run_in_executor()` | CPU/GPU-bound（PyTorch forward、数据预处理） | 并发度受线程数限制（默认 5×CPU），瓶颈在计算 |

**关键影响**：Nerva vs Triton 在 CPU mock 场景的 +64% QPS 优势主要来源于此——Triton Python backend 用 `time.sleep()`（阻塞线程），Nerva 用 `asyncio.sleep()`（非阻塞事件循环），单 Worker 多路复用能力差异显著。

**选型指南**：
- 模型 `infer()` 内部有 `await`（如 vLLM、HTTP 调用）→ `async_infer=True`
- 模型 `infer()` 是纯 CPU/GPU 同步计算 → `async_infer=False`
- 不确定时用 `False`（更安全，不会阻塞事件循环）

### 5.5 协作式取消

> cancel 是 best-effort + 最终可回收保证。

- **依据**：ADR-003
- **队列中请求**：收到 cancel → 立即移除 + 回收资源
- **执行中请求**：设 `context.cancelled=True` + `task.cancel()`
- **Backend 责任**：必须在推理循环中检查 `context.cancelled`，否则 cancel 不生效
- **状态返回优先级**：客户端 cancel → `ABORTED`；deadline 先触发 → `DEADLINE_EXCEEDED`；Worker 崩溃 → `UNAVAILABLE`

### 5.6 Worker 崩溃恢复

> 退避重启 + fail-fast 在途请求。

- **重启策略**：1s → 2s → 4s → ... → 16s，最多 5 次
- **在途请求**：崩溃时全部返回 `UNAVAILABLE`
- **启动失败**：回滚清理，不留半初始化 Worker
- **孤儿防护**：Worker 内置 watchdog，`os.getppid() != parent_pid` → 自行 SIGTERM

## 6. 性能原则

### 6.1 开销逐层可观测

> 每层独立埋点，支持瓶颈归因。

```
HTTP Request
  │  rpc_body_read_ms
  │  rpc_parse_ms
  ▼
[Executor.execute()]
  │  executor_scheduler_overhead_ms
  │
  ├─ [WorkerProxy.infer() × N stages]
  │    proxy_serialize_ms
  │    ipc_round_trip_ms / ipc_transport_ms
  │    proxy_deserialize_ms
  │
  │    [Worker._handle_infer()]
  │      worker_deser_ms
  │      backend_infer_ms
  │
  rpc_serialize_ms
```

每个时间点都可在 timing log 中追溯。排障路径：`request_id` 串联入口日志 → Executor 日志 → Worker 日志。

### 6.2 优化基于实测，不基于假设

> 每项优化决策都应有对应的 benchmark 数据支撑。

- 8KB inline 阈值 ← S1 spike 数据
- msgpack 选型 ← 序列化 < 0.1ms 实测
- PrecomputedGraph ← 框架开销分析中 Executor 调度开销量化
- **反例**：uvloop 在 macOS 上因 ZMQ 兼容性导致 p99 退化 11×，已回撤

### 6.3 尾延迟比中位数更重要

> 优化方向优先压 p99，而非 p50。

- 框架 p50 ~8ms，p99 可达 25ms+——差距主要来自 asyncio 调度抖动
- **主要尾延迟来源**（按优先级）：
  1. `rpc_body_read` p99=8.3ms（ASGI receive 调度等待）
  2. ZMQ `ipc_transport` p99=15ms（mm_postprocess 位置效应）
  3. `executor_scheduler` p99=3.4ms（Python asyncio.create_task 开销）

### 6.4 摊销一次性开销，最小化 per-request 开销

> 能在启动期算一次的，不要在每个请求里重复算。

| 一次性（per-pipeline） | per-request |
|------------------------|-------------|
| 拓扑排序 | ~~拓扑排序~~ |
| 邻接表构建 | ~~邻接表构建~~ |
| 输入策略推断 | ~~输入策略推断~~ |
| PrecomputedGraph 冻结 | Executor + in_degree 浅拷贝 |
| _PipelineExecutor 创建 | InferContext 创建 |

### 6.5 序列化在关键路径上越少越好

> 小 payload inline，大 payload SHM，特殊场景跳过 dict-level 编解码。

- ≤8KB：inline 进 ZeroMQ 消息，省 SHM alloc/free 开销
- >8KB：写入 SHM，控制消息只传 descriptor
- 单字段 bytes：`raw_bytes_v1` 跳过 dict-level `msgpack.packb`，Worker 侧用 `input_key` 还原
- SHM + msgpack 解码：优先 `memoryview → msgpack.unpackb`，避免 `bytes(buf[slice])` 中间副本

### 6.6 热路径预留 Rust/C++ 出口

> 当 Master 调度、序列化、数据拼接成为瓶颈时，offload 到 Rust/C++ 线程池。用户 API 不变。

- **Python-first 不等于 Python-only**——性能关键路径由框架内部优化（NumPy 模式）
- **候选热路径**：msgpack 编解码、frame 解析、DAG 调度循环、SHM 分配器
- **触发条件**：Master CPU 利用率在高并发下成为 QPS 天花板

## 7. 运行时原则

### 7.1 Fail-fast

> 任一关键节点失败 → 整条请求立即失败。不做重试，不做 partial success。

- **Executor**：节点异常 → break 循环 → cancel 所有 running tasks → 重抛
- **Pipeline 语义**：fail-fast 是 MVP 有意选择——重试和降级策略留给上层（网关或客户端）
- **理由**：框架只保证失败是快速且可诊断的

### 7.2 显式失败优于隐式等待

> 触发资源上限时走明确错误码，不做无界等待。

| 场景 | 行为 |
|------|------|
| SHM 满 | `RESOURCE_EXHAUSTED`，不阻塞等待 |
| 队列满 | 快速拒绝 `RESOURCE_EXHAUSTED` |
| Deadline 超期 | 立即丢弃，不进入推理 |
| Worker 不可用 | `UNAVAILABLE`，不静默排队 |

### 7.3 Deadline 从入口计时

> 客户端发绝对时间戳 → RpcHandler 转相对 TTL → 全链路用相对时间判断。

- 避免 Master/Worker 之间的时钟偏移问题
- batch 等待时间计入 deadline

### 7.4 request_id 全链路透传

> 从入口到 Backend 同一 ID，日志 + 指标均绑定。

- 排障时用 `request_id` 串联 RPC 日志 → Executor 日志 → Worker 日志
- **约束**：request_id 在 WorkerProxy._pending 中必须唯一，重复会覆盖已有 Future 导致 hang

### 7.5 错误码映射规范

> 异常到 RPC 错误码的映射应基于 typed exception，而非字符串匹配。

**当前实现**（待优化）：`"RESOURCE_EXHAUSTED" in str(exc)` 字符串匹配。

**目标状态**：引入 typed exception 层级：
```
NervaError
├── NervaDeadlineExceeded    → DEADLINE_EXCEEDED (4)
├── NervaResourceExhausted   → RESOURCE_EXHAUSTED (8)
├── NervaInvalidArgument     → INVALID_ARGUMENT (3)
└── NervaInternalError       → INTERNAL (13)
```

**过渡期约束**：改动异常消息时必须同步检查 `server/rpc.py` 的错误码映射逻辑。

错误码沿用 gRPC 子集，降低客户端重试逻辑的学习成本：
- `RESOURCE_EXHAUSTED` → 建议退避重试
- `DEADLINE_EXCEEDED` → 不重试（已超时）
- `INVALID_ARGUMENT` → 不重试（参数问题）
- `UNAVAILABLE` → 可重试（Worker 暂时不可用）

## 8. 工程原则

### 8.1 保守改动

> 宁可保守也不引入 bug。行为变更必须配套测试。

- 涉及并发、IPC、调度、资源回收的改动 → 必须补回归测试或最小复现脚本
- 不改动无关文件；发现无关脏改动 → 保持原样并单独说明
- 未经执行验证，不得声称"已修复""已通过"

### 8.2 不针对 mock artifact 改架构

> 优化设计必须遵守多进程解耦架构。mock 场景应贴近真实，不应针对 mock 特征定制。

- **反例**：为了压测数字好看，绕过 IPC 直接 in-process 调用 → 违反进程隔离原则
- **正例**：mock 模型用 asyncio.sleep 模拟延迟 + 真实 IPC 路径 → 测量的是真实框架开销

### 8.3 实测驱动决策

> IPC 方案选型、batching 参数、优化方向均基于实测数据而非假设。

- 新增优化 → 必须有 before/after 对比数据
- 优化回撤 → 记录原因和数据（如 uvloop 回撤）
- 性能声明 → 必须标注环境、配置、口径

## 9. API / 协议原则

### 9.1 instances 是部署配置，不是图结构

> 与 `backend`、`device`、`async_infer` 同层——放在 `model()` 声明中。

- 同一模型在不同部署环境可能需要不同 instances 数
- Graph/Node 结构不包含 instances 信息
- Executor 不感知多实例

### 9.2 版本不兼容即拒绝

> protocol version 不匹配 → `INVALID_ARGUMENT` + 关闭连接，不做隐式兼容。

- **理由**：隐式兼容是灰度期 bug 的温床
- 帧头 reserved 字段和 ext_hdr_len 为未来扩展预留空间

### 9.3 错误码复用 gRPC 子集

> 降低客户端重试逻辑的学习成本，复用业界通用语义。

## 10. 后续迭代规划项

以下是 MVP 有意推迟的能力，在后续迭代中评估实现：

| 项目 | 优先级 | 触发条件 |
|------|--------|---------|
| 请求间优先级（EDF） | 中 | 业务需要差异化 SLA |
| 优雅 drain（SIGTERM） | 中 | 生产部署需要零丢请求滚动更新 |
| Worker 侧入站背压 | 低 | 无 Batcher 场景下高并发导致 Worker OOM 或调度退化 |
| 图优化 pass（自动并行化、死节点消除） | 低 | 复杂 DAG 场景下用户手动优化成本高 |
| typed exception 替代字符串匹配 | 中 | 异常消息变更导致错误码误判 |
| 跨机器 Worker 调度 | 低 | 单机资源不满足吞吐需求 |
| 流式推理（server-streaming） | 高 | LLM 场景需要 TTFT 可观测 |

## 11. Review 检查清单

基于以上原则，review 代码变更时按以下维度检查：

### 架构一致性
- [ ] 是否违反多进程解耦？Master 进程内是否出现推理计算？
- [ ] 是否有 Executor 泄露 Worker 层细节？新增组件是否实现 InferableProxy？
- [ ] Pipeline 是否仍可独立执行（不依赖 HTTP 层）？

### DAG 正确性
- [ ] 图结构是否仍然静态不变？是否引入运行时图变更？
- [ ] Edge 的 src_field_path 和 dst_input_key 语义是否一致？
- [ ] cond/parallel 分支的闭包捕获是否正确生成跨图 Edge？
- [ ] 是否保留 R-PH2-PROXY-CAPTURE 回归用例？

### 调度正确性
- [ ] Deadline 三层守卫是否完整？新增路径是否检查 deadline？
- [ ] 背压是否仍在 Master 侧 Batcher？Worker 侧是否引入了隐式排队？
- [ ] 新增的 proxy 层是否对 Executor 透明？

### 性能影响
- [ ] 是否有 per-request 开销从 O(1) 退化到 O(N)？
- [ ] 是否有同步阻塞操作进入 asyncio 事件循环？
- [ ] 是否有实测数据支撑？before/after 对比？
- [ ] 改动是否影响 timing 埋点覆盖？

### 进程安全
- [ ] 多 uvicorn worker 场景是否安全？spawn-once / refcount 是否正确？
- [ ] 异常路径是否有资源泄漏（socket、SHM slot、lock file）？
- [ ] Worker 崩溃后在途请求是否有明确状态（UNAVAILABLE）？

### 错误处理
- [ ] 异常路径是否 fail-fast？是否有隐式等待或静默忽略？
- [ ] 错误码是否正确映射？新增异常是否遵循错误码规范？
- [ ] 资源耗尽是否有明确错误码（RESOURCE_EXHAUSTED）？

### 工程规范
- [ ] 行为变更是否配套测试？并发 / IPC 变更是否有回归测试？
- [ ] 是否针对 mock artifact 做了架构妥协？
- [ ] 文档是否同步更新（README、AGENTS.md §7/§8、roadmap）？
